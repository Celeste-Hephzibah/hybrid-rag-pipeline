Deploying a machine learning model into production involves more than achieving good accuracy on a held-out test set. Model serving infrastructure needs to handle request batching, latency requirements, and horizontal scaling under variable load. Common serving frameworks include TorchServe, TensorFlow Serving, and lightweight options like FastAPI wrapping a model directly for smaller-scale deployments such as a Streamlit Cloud demo.

Model drift occurs when the statistical properties of production data diverge from the data the model was trained on, causing performance to degrade silently over time. Monitoring pipelines typically track input feature distributions, prediction distributions, and, when ground truth becomes available with a delay, actual model accuracy over time. Alerting thresholds are usually set on these drift metrics so that a retraining pipeline can be triggered before performance degrades enough to cause business impact.

Vector databases such as FAISS, Pinecone, Weaviate, and Milvus are used to store and search dense embeddings at scale. FAISS, developed by Meta, is a library rather than a managed database, and is commonly used for local or self-hosted retrieval systems because it runs efficiently on a single machine without requiring external infrastructure. For production RAG systems handling millions of documents, a managed vector database with built-in sharding and replication is often preferred over a local FAISS index.

Continuous integration for machine learning, often abbreviated as CI/CD for ML or MLOps, extends traditional software CI/CD practices to include data validation, model training reproducibility, and automated evaluation against a held-out benchmark before a new model version is promoted to production. Feature stores are a related concept that centralize feature computation so that the same feature definitions are used consistently at both training time and inference time, preventing training-serving skew.
